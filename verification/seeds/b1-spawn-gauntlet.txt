# away horizontally. The task spec asks for seeds "covering hillside, cliff-edge, snowy-slope, and
# forest spawns" - these 20 values are an arbitrary fixed spread, NOT pre-classified by spawn
# terrain: this repo has no way to confirm a seed's spawn biome/terrain without actually
# generating the 1.21.1 world, so no biome claim is made here. Each run's actual spawn terrain
# (biome, local collision grid, fall-risk) is read from that run's own worldObservation telemetry
# after the fact. If the realized spread across these 20 turns out not to cover enough terrain
# diversity (e.g. everything lands in plains), swap in replacements and say so explicitly in the
# benchmark report - never assume diversity that wasn't actually observed.
#

# directly against level.dat NBT) - the terrain/generation contract a seed is supposed to pin is
# genuinely reproducible, this is not a Lodestone bug. But the exact spawn tile vanilla Minecraft's
# own first-join spawn-safety search settles the player on can still drift a few blocks between
# runs of the same seed - a base-game characteristic, not something this repo's UI automation
# controls. Since B1's waypoint is computed relative to the observed spawn tile (not baked into
# the seed), treat these 20 seeds as pinning reproducible terrain/biome, not a bit-for-bit
